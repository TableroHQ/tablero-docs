Service Breakdown

The current repository contains IdentityService, RestaurantService, ReservationService, OrderService, PaymentService, NotificationService, ApiGateway, SharedKernel, and Contracts. The final planned architecture keeps database-per-service isolation and adds DeliveryService as a dedicated microservice. Async communication uses Apache Kafka-ready contracts and local outbox publisher workers; real-time workflows use planned SignalR hubs.

Identity Service — :5001 · identity_db

RESPONSIBILITIES

  • Register user — email + username + password (PBKDF2 via ASP.NET Core Identity)
  • Login by email OR username on the same endpoint — field auto-detected
  • Issue JWT access tokens (15 min) + refresh tokens (7 days)
  • Refresh token rotation — old token invalidated on use, new pair issued
  • Update profile: name, avatar, phone, username
  • Password reset via email OTP — 6-digit code, 10 min expiry, single-use
  • Role assignment / change by Admin or Director
  • Account suspend / reactivate

TECHNICAL NOTES

  • JWT claims include: userId, email, username, role, restaurantId
  • Refresh tokens stored with device fingerprint (User-Agent hash)
  • Rate limiting on /auth/login via Polly + Redis counter
  • Avatar upload → stored as CDN URL (S3 / Azure Blob)
  • OTP stored in Redis — TTL 10 min, deleted on first use
  • Publishes user.registered and user.password.reset.requested events
Restaurant Service — :5002 · restaurant_db

RESPONSIBILITIES

  • Store restaurant info: name, address, geolocation (lat/lng), operating hours
  • Photo gallery management — upload, set primary, CDN-served URLs
  • Public menu display — no auth required (YARP bypass configured)
  • QR code generation per table → returns PNG or SVG
  • Aggregate ratings: chef avg, waiter avg, cleanliness avg, overall score
  • Operating hours + holiday schedule management
  • Multi-branch support — all data scoped by restaurant_id

TECHNICAL NOTES

  • Public endpoints bypass JWT — configured directly at YARP route level
  • QRCoder NuGet generates QR PNG encoding restaurantId + tableId
  • Ratings are denormalized aggregates — updated via review.submitted events from Notification Service
  • Menu displayed here is a read-only cache — canonical menu lives in Order Service, synced via menu.item.updated events
  • Photos stored as URL references; binary data served from CDN
Reservation Service — :5003 · reservation_db

RESPONSIBILITIES

  • Table layout management — indoor / outdoor / bar / terrace zones
  • Time slot configuration per restaurant, managed by Admin
  • Availability check with Redis distributed lock preventing double-booking
  • Reservation creation with exact date + time slot + party size
  • Pre-order food items with reservation — pushed to kitchen on seating
  • Status lifecycle: PENDING → CONFIRMED → SEATED → COMPLETED / NO_SHOW / CANCELLED
  • Automated reminder 24h before via Quartz.NET scheduler

TECHNICAL NOTES

  • Redis lock key pattern: slot:{restaurantId}:{tableId}:{date}:{slotId} — TTL 30 sec during booking transaction
  • Pre-order items stored in reservation_pre_orders table — forwarded to Order & Kitchen Service when customer is seated via reservation.seated event
  • QR scan populates X-Table-Id header forwarded through YARP to this service
  • Publishes: reservation.confirmed, reservation.seated, reservation.cancelled, reservation.completed, reservation.no_show, reservation.reminder
Order & Kitchen Service — :5004 · order_db

RESPONSIBILITIES

  • Canonical menu management — categories, items, modifiers, pricing, allergens
  • Order placement by customer from app or waiter on behalf of customer
  • Orders scoped to: table session, delivery, or pre-order
  • Kitchen Display System (KDS) — real-time order queue for chefs via SignalR
  • Audio chime + visual alert on new order (browser Audio API)
  • Item status: QUEUED → PREPARING → READY
  • Chef calls specific waiter for table pickup via WaiterHub
  • Kitchen printer webhook — POST order ticket to configurable endpoint
  • Mark item unavailable (86'd) — reflected to public menu in real-time via event

TECHNICAL NOTES

  • KitchenHub: chefs join group kitchen:{restaurantId}
  • WaiterHub: waiters join group waiter:{waiterId}
  • TableHub: customers join group table:{tableId} for order status
  • Kitchen printer: configurable webhook URL per restaurant; payload ZPL or ESC/POS JSON
  • Order items store price_snapshot — price at time of order, immutable after placement
  • Publishes: order.placed, order.item.ready, order.ready, order.served, order.cancelled, menu.item.unavailable
Payment & Loyalty Service — :5005 · payment_db

RESPONSIBILITIES

  • Invoice generation from completed orders — triggered by order.ready event
  • Stripe Checkout for card payments
  • Cash payment recorded by cashier via dedicated POS app
  • Split bill across multiple customers at same table
  • Account balance management — top-up + deductions
  • Loyalty: earn X points per currency unit spent
  • Loyalty: bonus catalogue — food items redeemable for points
  • Bonus redemption → zero-price line item on invoice
  • Delivery pre-payment — balance held, released on delivery confirmation

TECHNICAL NOTES

  • Stripe webhooks handled: payment_intent.succeeded, payment_intent.payment_failed, charge.refunded
  • Delivery balance: hold (not deduction) → confirmed on delivery receipt → or auto-refund after 2h if unconfirmed
  • Bonus points: calculated after payment.succeeded event — points_earned = floor(total × rate)
  • Cashier POS: dedicated minimal app uses /cashier/* endpoints only
  • Publishes: payment.succeeded, payment.failed, payment.cash.recorded, invoice.generated, bonus.earned, bonus.redeemed
Delivery Service — :5006 · delivery_db · planned

RESPONSIBILITIES

  • Delivery order creation — triggers full pre-payment via Payment Service
  • Broadcast to all available online couriers: SignalR push + Twilio SMS
  • First-accept assignment — race condition handled via Redis lock
  • Courier GPS checkpoint recording — lat/lng + timestamp per checkpoint
  • Customer confirms receipt in app → releases balance hold
  • Auto-confirm + balance release after 2 hours if no customer action
  • Customer rates courier (1–5 stars + comment)
  • Courier average rating computed and displayed on profile

TECHNICAL NOTES

  • Courier availability set via ONLINE / OFFLINE toggle in courier app
  • First-accept lock key: delivery:{deliveryId}:accept — TTL 5 sec
  • GPS checkpoints: last N positions shown to customer, full history retained
  • CourierHub: assignment broadcasts to group couriers:{restaurantId}
  • Publishes: delivery.broadcast, delivery.assigned, delivery.checkpoint, delivery.confirmed, delivery.rated
Notification Service — :5007 · notify_db · review features planned

RESPONSIBILITIES

  • Consumes ALL platform events → dispatches appropriate notifications
  • Email via SendGrid: receipts, OTPs, reminders, promotions
  • SMS via Twilio: reservation reminders, delivery updates
  • In-app push via SignalR NotifyHub + FCM for mobile
  • Review submission: chef, waiter, cleanliness, service ratings + comment
  • Review moderation — Admin approve / reject before publication
  • Aggregate ratings → publish review.submitted event → Restaurant Service updates

TECHNICAL NOTES

  • Review only allowed after COMPLETED reservation or confirmed delivery — enforced server-side
  • One review per reservation / delivery per customer — enforced by unique DB constraint
  • Promotional emails: bulk via SendGrid Marketing API, scoped per restaurant
  • Chef/waiter ratings stored per staff member — average computed on query (Dapper)
  • Notification preferences: per-customer opt-out of SMS / email per category
  • Audit log: all sent notifications stored with delivery status (sent / failed / bounced)

DATA ARCHITECTURE

Database Design

Isolated PostgreSQL databases — one per implemented or planned microservice. No cross-service foreign keys. Cross-service references are stored as UUID columns only. Services that publish integration events own an outbox_events table for guaranteed event delivery via Transactional Outbox.

Design Principles
DecisionRationale
Database-per-serviceEach service owns its PostgreSQL database exclusively. Data sharing only via events or REST — never direct cross-service DB access.
UUID primary keysNo auto-increment leaking row counts. Safe for distributed systems. Generated via gen_random_uuid().
TIMESTAMPTZ everywhereSystem spans multiple cities/countries. Timezone-aware timestamps prevent silent data corruption.
Enums as VARCHARReadable in queries. No enum type migration headaches. CHECK constraints enforce valid values.
Soft delete (deleted_at)Menu items, reservations, orders, users use deleted_at timestamps. EF Core global query filters exclude them automatically.
Transactional OutboxEvery service has an outbox_events table. Business data and events are written in the same EF Core transaction. outbox publishers read pending events and publish to Kafka.
No cross-DB foreign keysCross-service references store the UUID and rely on application-level validation via REST calls or eventual consistency via events.
Schema annotations:    PK — primary key FK — foreign key (same DB) xref — cross-service UUID reference (no FK constraint)

Tablero — Restaurant Operating System

A full-stack platform connecting every actor in the restaurant lifecycle — from QR menus and table reservations to real-time kitchen displays, delivery tracking, loyalty rewards, and cashier POS. The current repository includes six .NET 10 / ASP.NET Core Web API services plus the YARP API Gateway, SharedKernel, and Contracts projects. The final planned architecture adds Delivery as a dedicated service, with database-per-service isolation and Kafka-ready outbox events.

Vision
What you're building is not a reservation app — it's a full Restaurant Operating System. It ties together every participant in a restaurant in real time: guests browsing before arrival, customers ordering from the table, chefs watching a live kitchen queue, waiters managing service from their phones, cashiers processing payments, couriers delivering food, and directors running multi-branch operations. This is Square + OpenTable + Uber Eats + a loyalty program — unified under one coherent platform.

CORE PROBLEM

No single system ties together reservations, in-restaurant ordering, kitchen workflow, payments, delivery, and customer loyalty. Tablero solves this in one platform.

KEY DIFFERENTIATOR

Food-item bonuses — not just points. Earning a real hotdog or burger from spending makes loyalty feel on-brand and memorable for a food platform.

TECHNICAL CHALLENGE

Chef ↔ Waiter ↔ Customer synchronization requires SignalR hubs on top of Kafka events. Kitchen printer integration adds IoT-layer complexity.

BUSINESS MODEL

Restaurants subscribe per branch. All data is scoped by restaurant_id. Multi-branch support allows the platform to scale across chains without re-architecture.

COMPLIANCE

Card payments delegated entirely to Stripe (PCI-DSS). Customer profiles and review data require GDPR-compliant data handling and deletion flows.

SCOPE ASSESSMENT

Recommended 3-phase delivery: Auth + Menu → Orders + Kitchen + Payments → Delivery + Loyalty + Reviews.

Architecture

STYLE

Current implementation: IdentityService, RestaurantService, ReservationService, OrderService, PaymentService, NotificationService, ApiGateway, SharedKernel, and Contracts. Final planned architecture adds DeliveryService as a separate microservice. No shared database access.

COMMUNICATION

Async: Kafka via Transactional Outbox + Outbox publisher
Sync: REST/HTTP via Refit + Polly circuit breakers
Real-time: SignalR (planned) (6 hubs)

AUTH

Gateway-centralized JWT. Token validated once at YARP Reverse Proxy, user identity forwarded as trusted headers (X-User-Id, X-User-Role, X-User-Email, X-Restaurant-Id) to downstream services.

INFRASTRUCTURE

  • YARP Reverse Proxy (API Gateway + JWT validation)
  • Service discovery (planned)
  • Apache Kafka + Outbox workers (Messaging)
  • OpenTelemetry + Grafana Tempo + Loki (Observability)

DATA & INTEGRATIONS

  • PostgreSQL (one per service)
  • Redis (planned: distributed locks, OTP cache, rate limiting)
  • Stripe (Checkout + Webhooks)
  • SendGrid (email) + Twilio (SMS)

OBSERVABILITY

  • OpenTelemetry (instrumentation)
  • Grafana Tempo (distributed traces)
  • Grafana Loki + Promtail (log aggregation)
  • Grafana (unified dashboards)

Key Features

AUTHENTICATION

  • Login by email OR username — single field, auto-detect by presence of "@"
  • JWT access token (15 min) + refresh token (7 days) with rotation
  • Password reset via email OTP — 6-digit code, Redis TTL 10 min, single-use
  • Roles encoded in JWT claims alongside userId, restaurantId

LOYALTY SYSTEM

  • Points earned per currency unit spent — rate configurable by Director
  • Bonus catalogue: food items (Hotdog = 50pts, Burger = 120pts) — configured per restaurant
  • Redemption creates a zero-price line item on the next invoice
  • Points stored in user_balances.loyalty_points

KITCHEN DISPLAY SYSTEM

  • Chef screen — live order queue via KitchenHub SignalR
  • New order triggers audio chime + animated card
  • Item lifecycle: QUEUED → PREPARING → READY
  • Chef calls specific waiter → WaiterHub push notification
  • Optional kitchen printer webhook — ZPL/ESC-POS JSON payload

DELIVERY SYSTEM

  • 100% balance pre-payment — held (not deducted) until delivery confirmed
  • Broadcast to all online couriers via SignalR + Twilio SMS
  • First-accept assignment via Redis lock delivery:{id}:accept TTL 5 sec
  • GPS checkpoints — lat/lng + timestamp visible to customer
  • Auto-confirm + balance deduction after 2h if no customer action

QR CODE ORDERING

  • QR encodes restaurantId + tableId — unique per table
  • Public menu accessible without login (configured at YARP level)
  • Authenticated scan auto-assigns user to table session
  • QR codes generated as PNG/SVG via QRCoder NuGet

PAYMENT FLOWS

  • App payment: Stripe Checkout or account balance
  • Cash: customer → waiter → cashier marks paid in POS → CashierHub notifies
  • Split bill: multiple customers per table, each pays their share
  • Receipt emailed automatically via SendGrid after payment event

Key Architectural Decisions
Login field auto-detection: Single input — if value contains "@", route to email lookup; otherwise username lookup. Both map to the same user record in users table.

Delivery pre-payment hold: Balance is held (not transferred) at order placement. Converted to deduction when customer confirms receipt. Auto-released after 2h if unconfirmed. Refunded if cancelled before kitchen pickup.

Kitchen printer as webhook: Order Service POSTs to a configurable endpoint on order.placed. Optional per restaurant, ZPL or ESC-POS JSON payload.

Cashier as distinct role: Cashier gets a minimal dedicated POS app — only sees open tables, bills pending payment, cash marking, and table release. No kitchen, menu management, or reporting access.

Multi-restaurant scoping: Every entity is scoped by restaurant_id. A Director account owns multiple restaurant branches. Admin is scoped to one branch.

LEVEL 2

Container Diagram

Zooms inside Tablero to show its containers — the deployable units and data stores that make up the system. Each microservice is an independently deployable .NET 10 / ASP.NET Core application with its own PostgreSQL database. Services communicate asynchronously via Kafka events and in real-time via SignalR WebSocket hubs.

WHAT THIS DIAGRAM SHOWS

The deployable pieces inside the system boundary: YARP API Gateway, six implemented ASP.NET Core Web API services, SharedKernel and Contracts projects, plus a planned Delivery Service. The target architecture keeps one PostgreSQL database per domain service, Apache Kafka with Transactional Outbox + outbox publishers, Redis, service discovery, 6 SignalR hubs, and OpenTelemetry + Grafana observability.

KEY ARCHITECTURAL DECISION

Microservices with Database-Per-Service. Each service owns its data exclusively — no shared databases. Inter-service communication uses Kafka (via Transactional Outbox + Outbox publisher) for async events and REST/HTTP with Polly + Refit circuit breakers for synchronous queries. YARP Reverse Proxy performs centralized JWT validation and forwards identity headers (X-User-Id, X-User-Role, X-User-Email, X-Restaurant-Id) — downstream services never parse JWTs. 6 SignalR hubs provide real-time WebSocket communication for kitchen, waiter, table, courier, cashier, and notification channels.

LEVEL 3

Component Diagram

Zooms inside each microservice to show its internal components. Each service is an .NET 10 / ASP.NET Core application with its own controllers, services, repositories, and domain entities. Services communicate through REST/HTTP for synchronous queries and Kafka events for async notifications. No service directly accesses another service's database.

MICROSERVICE RULES

1. Each service owns its own database — no shared DB access. 2. Synchronous cross-service calls use REST/HTTP via Refit with Polly circuit breakers. 3. Async events go through the Transactional Outbox pattern (write to outbox table → Outbox publisher → Kafka). All consumers are idempotent with Dead Letter Topics. 4. YARP Gateway validates JWT centrally and forwards X-User-Id, X-User-Role, X-User-Email, X-Restaurant-Id headers — downstream services just read headers.

WHY THIS STRUCTURE

Each microservice is independently deployable, scalable, and maintainable. Order & Kitchen Service combines menu management and kitchen operations into one bounded context (canonical menu + order lifecycle). Payment & Loyalty Service combines invoicing and loyalty — bonus redemption creates a zero-price line item on the same invoice. Database-per-service ensures strong data ownership.

LEVEL 4

Code — Entity Diagram

The deepest level of detail. Shows the key domain entities within each service's database — the tables, their primary fields, and how they relate. Each service owns its own PostgreSQL database; cross-service references use denormalized IDs (UUIDs), never foreign keys.

CONVENTIONS

All primary keys are UUIDs. Timestamps are timestamptz. Every database has an outbox_events table for the Transactional Outbox pattern. Cross-service IDs are stored as plain UUIDs with no FK constraint. Soft deletes where applicable.

ORM STRATEGY

EF Core 10 for writes and migrations. Dapper for complex read queries (aggregations, reports, joins). PostgreSQL with JSONB support for flexible metadata columns.
